1. Executive Summary

MyHigh5 is a multilingual contest and voting platform built on FastAPI (Python) for the backend and Next.js 14 (TypeScript/React) for the frontend, backed by PostgreSQL and Redis. The platform integrates cryptocurrency payments (BSC, Ethereum, Solana, Bitcoin), KYC identity verification (Shufti Pro), a multi-level affiliate system, social features, and a CPA-compliant double-entry accounting engine.

This assessment was conducted as a white-box static code review. The codebase is large (~36 database models, 45+ API endpoint files, ~250 KB of route code) and generally follows good practices such as Pydantic input validation, bcrypt password hashing (12 rounds), parameterised ORM queries via SQLAlchemy, and HMAC-based webhook signature verification.

However, several critical and high-severity issues were identified — most of them centred on secret management, token lifecycle, and file-upload validation. Two of the critical issues (hardcoded JWT secret and committed production secrets) could allow an attacker to forge authentication tokens or directly compromise third-party service accounts without needing to exploit any runtime vulnerability.

Overall Risk Rating: HIGH — Immediate action is required on the two CRITICAL findings before this application is operated in a production environment.

F-01 · Hardcoded Default JWT Secret Key

IDF-01
SeverityCritical
OWASP CategoryA02:2021 – Cryptographic Failures
Locationbackend/app/core/config.py – line ~34
CVSS Score (Est.)9.8 (Critical)

Description

The application's configuration class defines a fallback value for SECRET_KEY — the signing key used for all JWT authentication tokens:

SECRET_KEY: str = "c7d9fbef58c5ac4fe7d31e32a89e2c336468843c1c307177ab0ee55f54063115"

If the production environment does not set the SECRET_KEY environment variable, this static, publicly visible key is used. Any attacker who reads the source code (or this report) can forge valid JWT tokens for any user, including administrators, without knowing a password. Because the token is the only authentication mechanism, this effectively bypasses the entire authentication layer.

Impact

  • Complete authentication bypass – attacker can sign a JWT with sub=<any_user_id>
  • Full privilege escalation to administrator
  • Access to all financial, KYC, and personal data
  • Ability to approve KYC, transfer funds, modify contests

Proof of Concept (Conceptual)

# An attacker who knows the secret key can produce:
import jwt, time
payload = {"sub": "1", "exp": int(time.time()) + 86400*7}
token = jwt.encode(payload, "c7d9fbef58c5ac4fe7d31e32a89e2c336468843c1c307177ab0ee55f54063115", algorithm="HS256")
# Use token as Bearer token to authenticate as user ID 1 (typically admin)
How to Fix
  • Remove the hardcoded fallback value entirely; make the setting required:
    SECRET_KEY: str # No default — will raise at startup if not set
  • Generate a cryptographically random key of at least 256 bits:
    python -c "import secrets; print(secrets.token_hex(32))"
  • Store the value exclusively in the server's environment (AWS Secrets Manager, Vault, or a secure CI/CD secret).
  • Add a startup assertion that fails if the key is shorter than 32 bytes or matches known defaults.
  • Rotate the key immediately in all existing environments; invalidate all active tokens by doing so.

IDF-02
SeverityCritical
OWASP CategoryA02:2021 – Cryptographic Failures / A05:2021 – Security Misconfiguration
Locationfrontend/.env.production
CVSS Score (Est.)9.1 (Critical)

Description

The frontend/.env.production file has been committed to the git repository and contains live production secrets:

# Found in frontend/.env.production (committed to source control)
CURSOR_API_KEY=crsr_8cced7dbd01f51b958646d22528928eb505ca38b57849d6dddb770ecb0783a3d
MAINTENANCE_BYPASS_TOKEN=3554c74e6eb44910a1c19808b5531abef434b4ddf4d74d8596af0f1978fc2c18
NEXT_PUBLIC_REOWN_PROJECT_ID=8fdb507d78ee0a3b9df5355281be94a1
NEXT_PUBLIC_ANNUALADS_API_KEY=d9de1649ceec6f9efd7b3a9154ae83a9f5d40a1feb4ff5ee2487ec5a2331e56d

Even if this repository is private today, leaked credentials in git history are permanent — any repository exposure (insider threat, misconfigured deployment, future public release) makes these secrets immediately usable by attackers.

Impact

  • The MAINTENANCE_BYPASS_TOKEN allows any attacker to access the application while it is in maintenance mode, bypassing the intended service interruption protection.
  • The ANNUALADS_API_KEY could allow unauthorised advertising API calls or financial fraud via the sponsor integration.
  • The CURSOR_API_KEY exposes a paid AI translation service to abuse.

  • Remove the files from git tracking: git rm --cached frontend/.env.production
  • Purge the secrets from git history using git filter-repo or BFG Repo Cleaner.
  • Immediately rotate every exposed secret — treat them as compromised.
  • Use platform-level environment variable injection (Vercel Dashboard, GitHub Actions Secrets, AWS Parameter Store) instead of committing .env files.
  • Implement a pre-commit hook (e.g., detect-secrets) to prevent future secret commits.

F-03 · No JWT Refresh / Token Revocation Mechanism

IDF-03
SeverityHigh
OWASP CategoryA07:2021 – Identification and Authentication Failures
Locationbackend/app/core/security.py, backend/app/api/endpoints/auth.py

Description

The platform uses stateless JWT tokens with a 7-day expiry (10,080 minutes) and provides no token revocation endpoint (logout), no refresh-token flow, and no token blocklist. Once a token is issued, it is valid for 7 days regardless of:

  • User password changes
  • User account suspension or banning
  • Suspected token theft
  • User-initiated logout

Notably, there is a ban mechanism in the admin endpoints, but a banned user's existing token continues to work until it expires naturally — a 7-day window of continued access.

Impact

  • Stolen tokens remain usable for up to 7 days after theft is detected.
  • Users cannot truly log out; clearing client storage does not invalidate the server-side token.
  • Banned or suspended users retain full API access for the remainder of the token lifetime.
How to Fix
  • Implement a token blocklist in Redis: on logout or ban, store the token's JTI (JWT ID claim) with a TTL matching the token's remaining lifetime. Check the blocklist on every authenticated request.
  • Alternatively, use short-lived access tokens (15 minutes) combined with long-lived refresh tokens (7 days, single-use, stored in HttpOnly cookies). Refresh tokens can be revoked server-side.
  • Add a jti (JWT ID) claim to every token so individual tokens can be targeted for revocation.
  • The get_current_active_user() dependency already checks is_active — ensure this check is always hit and that banning sets is_active=False atomically.

F-04 · Unrestricted File-Upload MIME Type Spoofing

IDF-04
SeverityHigh
OWASP CategoryA03:2021 – Injection / A04:2021 – Insecure Design
Locationbackend/app/api/endpoints/media.py – upload endpoint

Description

The upload endpoint validates the file's Content-Type header (must start with image/ or video/). However, the Content-Type is attacker-controlled — it is sent by the client and is not verified against the actual file magic bytes. A malicious actor can:

curl -X POST /api/v1/media/upload \
  -H "Authorization: Bearer <token>" \
  -F "file=@malware.php;type=image/jpeg"

The backend will accept the file as a valid image. Depending on the web server configuration, served files could be interpreted as executable scripts.

Impact

  • Uploading server-side scripts (PHP, Python, shell scripts) disguised as images.
  • Potential Remote Code Execution (RCE) if the server executes the uploaded file.
  • Malware distribution via the platform's CDN/media serving infrastructure.
How to Fix
  • Validate file content using magic bytes / file signatures (not just the Content-Type header). Use Python's python-magic library:
    import magic; mime = magic.from_buffer(file_content, mime=True)
  • Use Pillow's Image.open() + image.verify() to confirm image files are truly valid images — this will reject non-image data.
  • Strip EXIF metadata from uploaded images using Pillow to prevent metadata injection.
  • When serving uploaded files, set Content-Disposition: attachment and X-Content-Type-Options: nosniff to prevent browser execution.
  • Serve all user uploads from a separate domain (not the main app domain) to isolate any XSS or script-execution risk.
  • If using S3, never make the bucket publicly readable; use pre-signed URLs with short expiry.

F-05 · No Upload File-Size Limit Enforced

IDF-05
SeverityHigh
OWASP CategoryA04:2021 – Insecure Design
Locationbackend/app/api/endpoints/media.py

Description

No explicit maximum file size is enforced in the upload endpoint or in the FastAPI/uvicorn configuration. While the Contest model has optional fields for media_max_size_mb and video_max_duration_seconds, these are not applied as upload guardrails at the API level. An authenticated user can upload arbitrarily large files, consuming storage and memory resources.

Impact

  • Storage exhaustion on local disk or excessive S3 costs.
  • Memory exhaustion on the server (reading large files into memory during processing).
  • Application-level Denial of Service for other concurrent users.
How to Fix
  • Enforce a global maximum upload size in FastAPI:
    app = FastAPI() # then add middleware or use max_upload_size in multipart handling
  • Configure uvicorn with a maximum request body size: --limit-max-requests / set MAX_REQUEST_SIZE in a custom middleware.
  • Read uploaded files in chunks rather than loading the entire file into memory.
  • Apply per-contest size restrictions from the Contest model as hard limits at the API level, not just as advisory metadata.
  • Set S3 multi-part upload conditions with pre-signed URL expiry and maximum byte constraints.

F-06 · CORS Wildcard Allow-Headers

IDF-06
SeverityHigh
OWASP CategoryA05:2021 – Security Misconfiguration
Locationbackend/main.py – CORSMiddleware configuration

Description

The CORS middleware is configured with:

allow_headers=["*"]
allow_credentials=True
expose_headers=["*"]

Combining allow_headers=["*"] with allow_credentials=True is prohibited by the CORS specification in modern browsers and creates a security risk. Some browsers may ignore the wildcard when credentials are enabled, but this configuration signals a lack of intentional header restriction. Additionally, the custom CORSExtraMiddleware uses regex matching that may inadvertently allow unintended origins.

Impact

  • Cross-origin requests from attacker-controlled sites may include sensitive headers.
  • Any subdomain compromise (XSS on a Vercel preview deploy) could make credentialed cross-origin requests to the API.
  • Regex origin matching errors could whitelist unintended origins.
How to Fix
  • Replace allow_headers=["*"] with an explicit allowlist:
    allow_headers=["Authorization", "Content-Type", "Accept", "X-Requested-With"]
  • Replace expose_headers=["*"] with only the headers the frontend actually reads.
  • Audit the regex in CORSExtraMiddleware — use exact string matching where possible, and test edge cases (e.g., evilmyhigh5.com matching a pattern intended for myhigh5.com).
  • In production, ensure the allowed origin list does not include localhost or preview deployment wildcards.

F-07 · User Enumeration via Email / Username Probing

IDF-07
SeverityMedium
OWASP CategoryA07:2021 – Identification and Authentication Failures
Locationbackend/app/api/endpoints/auth.py – login, password-reset endpoints

Description

The login endpoint may return different error messages or response codes depending on whether the email/username exists versus whether the password is wrong. The registration endpoint likely returns a distinct error when an email is already taken. The password-reset endpoint may confirm whether an account exists for a given email. This allows an attacker to enumerate valid user accounts through response differences.

Impact

  • Attacker builds a list of valid platform usernames/emails for targeted phishing or credential stuffing.
  • Facilitates social engineering attacks against identified users.
How to Fix
  • Return identical generic error messages for all authentication failures: "Invalid credentials" (not "User not found" vs "Wrong password").
  • The password-reset endpoint should always return the same response regardless of whether the email exists: "If an account with that email exists, you will receive a reset link."
  • Add a constant-time delay to authentication responses to prevent timing-based enumeration.
  • Implement CAPTCHA on registration and password-reset to slow automated probing.

F-08 · No Rate Limiting on Authentication Endpoints

IDF-08
SeverityMedium
OWASP CategoryA07:2021 – Identification and Authentication Failures
Locationbackend/app/api/endpoints/auth.py, backend/main.py

Description

No rate-limiting middleware or decorator was observed on any API endpoint, including POST /auth/login, POST /auth/register, POST /auth/password-reset-request, and voting endpoints. While Redis is available in the stack, it is not used for request throttling. An attacker can make unlimited requests to these endpoints.

Impact

  • Brute-force attacks against user passwords (login endpoint).
  • Mass automated account registration (spam/bot accounts).
  • Email bombing via repeated password-reset requests.
  • Vote manipulation through automated voting attempts.
How to Fix
  • Implement rate limiting using slowapi (the FastAPI-native rate limiter built on Redis):
    pip install slowapi – then decorate sensitive endpoints with @limiter.limit("5/minute").
  • Apply stricter limits to authentication endpoints (e.g., 5 login attempts per IP per 15 minutes with exponential backoff).
  • Use Redis to track failed login attempts per IP and per username; implement account lockout after N failures.
  • Consider deploying a WAF (Cloudflare, AWS WAF) in front of the API for additional layer protection.

F-09 · Verbose Swagger / OpenAPI Documentation Exposed

IDF-09
SeverityMedium
OWASP CategoryA05:2021 – Security Misconfiguration
Locationbackend/main.py – FastAPI default docs at /docs and /redoc

Description

FastAPI automatically exposes interactive Swagger UI at /docs and ReDoc at /redoc. The OpenAPI JSON schema is available at /openapi.json. These endpoints are currently unrestricted and provide a complete map of all API routes, request/response schemas, authentication requirements, and example payloads — serving as a ready-made attack surface guide for any attacker.

Impact

  • Full API reconnaissance without any authentication.
  • Attacker can identify admin-only endpoints and target them specifically.
  • Internal schema and field names revealed, aiding injection attack crafting.
How to Fix

  • Alternatively, protect the docs behind authentication or IP allowlisting (restrict to office IPs / VPN).
  • Use an environment variable to toggle docs on/off:
    docs_url="/docs" if settings.DEBUG else None

F-10 · HS256 JWT Algorithm (Shared Secret) vs RS256

IDF-10
SeverityMedium
OWASP CategoryA02:2021 – Cryptographic Failures
Locationbackend/app/core/config.py, backend/app/core/security.py

Description

The JWT tokens use HMAC-SHA256 (HS256) which requires the same secret key for both signing and verification. In a microservices architecture (the codebase has a /auth/validate-token endpoint suggesting microservice token validation), this means every service that validates tokens must have a copy of the secret — increasing the attack surface. If any service is compromised, the shared secret allows forgery of tokens for all services.

Impact

  • Single secret compromise → full token forgery across all services.
  • Secret must be shared with all token-validating services, multiplying exposure.
How to Fix
  • Migrate to RS256 (RSA asymmetric signing): the auth service holds the private key for signing; all other services hold only the public key for verification.
  • This way, only the auth service can mint tokens; other services cannot.
  • Generate a 2048-bit or 4096-bit RSA key pair and expose the public key at a JWKS endpoint (/.well-known/jwks.json).

F-11 · Overly Long JWT Token Expiry (7 Days, No Rotation)

IDF-11
SeverityMedium
OWASP CategoryA07:2021 – Identification and Authentication Failures
Locationbackend/app/core/config.py – ACCESS_TOKEN_EXPIRE_MINUTES = 10080

Description

Access tokens are valid for 7 days (10,080 minutes) with no rotation mechanism. This is an unusually long expiry for a high-value platform dealing with financial transactions, KYC data, and cryptocurrency wallets. Token theft (via XSS, network interception, or device compromise) gives an attacker a 7-day window of authenticated access.

Impact

  • Stolen token usable for 7 days — ample time for financial fraud.
  • No way for users to invalidate tokens without waiting for expiry.
How to Fix
  • Reduce access token expiry to 15–60 minutes.
  • Implement a secure refresh token flow (see F-03) with refresh tokens stored in HttpOnly, Secure, SameSite=Strict cookies.
  • Implement sliding session expiry: issue a new access token on each API call if the existing token is within the last 5 minutes of validity.

F-12 · Insecure Default Encryption Salt

IDF-12
SeverityMedium
OWASP CategoryA02:2021 – Cryptographic Failures
Locationbackend/app/core/config.py – ENCRYPTION_KEY_DERIVATION_SALT

Description

The encryption key derivation salt used for end-to-end messaging has a hardcoded default value:

ENCRYPTION_KEY_DERIVATION_SALT: str = "default-salt-change-in-production"

If this value is not overridden in production, all derived encryption keys for private messages share a predictable salt, significantly weakening the key derivation function (KDF) and making key derivation attacks feasible.

Impact

  • Predictable key derivation may allow decryption of private messages if the master key is also compromised.
  • Salt collision across deployments if multiple instances share the same default.
How to Fix
  • Remove the default value and make it a required environment variable.
  • Generate a random 32-byte salt: python -c "import secrets; print(secrets.token_hex(32))"
  • Add a startup check that rejects the known default value and fails loudly.
  • Use per-user or per-conversation salts in addition to the global salt for better key isolation.

F-13 · Missing Security HTTP Headers

IDF-13
SeverityMedium
OWASP CategoryA05:2021 – Security Misconfiguration
Locationbackend/main.py (middleware config), frontend/next.config.js

Description

The application does not explicitly configure the following security response headers on API responses or frontend pages:

  • Strict-Transport-Security (HSTS) – enforce HTTPS
  • X-Content-Type-Options: nosniff – prevent MIME-sniffing
  • X-Frame-Options: DENY – prevent clickjacking
  • Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy – restrict browser features
  • Content-Security-Policy (see F-19)
How to Fix
  • Add a SecurityHeadersMiddleware to FastAPI that injects these headers on all responses.
  • In next.config.js, add the headers() function to set security headers for all Next.js routes.
  • Use securityheaders.com to verify headers after deployment.
  • Set Strict-Transport-Security: max-age=31536000; includeSubDomains; preload after confirming full HTTPS deployment.

F-14 · S3 Bucket Object Enumeration Risk

IDF-14
SeverityMedium
OWASP CategoryA01:2021 – Broken Access Control
Locationbackend/app/api/endpoints/media.py – media serving endpoint

Description

The media serving endpoint GET /api/v1/media/file/{user_id}/{filename} serves files based on a predictable user_id/filename path. If the S3 bucket is publicly accessible, or if filenames follow a predictable pattern, an attacker can enumerate and access media files belonging to other users — including KYC documents — by iterating user IDs.

Impact

  • Unauthorised access to other users' profile images, contest media, and potentially KYC documents.
  • Privacy violation and potential regulatory (GDPR/PIPEDA) non-compliance.
How to Fix
  • Require authentication on the media serving endpoint and verify the requesting user owns or has permission to view the requested file.
  • For S3: keep the bucket private and generate pre-signed URLs (15–60 minute expiry) rather than serving files through the backend.
  • Use cryptographically random filenames (UUID4 or similar) to prevent predictable enumeration.
  • KYC documents must be served only to the owning user or authorised admin staff, never publicly.

F-17 · Circular Role Inheritance — Potential DoS

IDF-17
SeverityLow
OWASP CategoryA04:2021 – Insecure Design
Locationbackend/app/models/role.py – get_all_permissions()

Description

The Role model supports hierarchical inheritance via a self-referential foreign key (parent_id). The get_all_permissions() method recursively traverses the inheritance chain. If an admin accidentally creates a circular role chain (Role A → Role B → Role A), this method will recurse infinitely and crash the worker process — a self-inflicted Denial of Service on any authenticated request.

How to Fix
  • Add cycle detection to get_all_permissions() using a visited set:
    def get_all_permissions(self, visited=None): if visited is None: visited = set(); if self.id in visited: return set(); visited.add(self.id); ...
  • Add a database constraint or admin-level validation to prevent creating circular role chains at insert/update time.

F-18 · Login Log IP Address Spoofable

IDF-18
SeverityLow
OWASP CategoryA09:2021 – Security Logging and Monitoring Failures
Locationbackend/app/models/login_log.py, auth endpoint

Description

The LoginLog model stores the user's IP address, which is read from request headers. If the application is behind a reverse proxy and trusts headers like X-Forwarded-For or X-Real-IP without validation, an attacker can spoof their IP address by injecting these headers, making security auditing and IP-based blocking unreliable.

How to Fix
  • Configure the application to trust the proxy header (X-Forwarded-For) only when the request comes from a known trusted proxy IP. Use trusted_hosts middleware or configure uvicorn with --forwarded-allow-ips.
  • Use Starlette's ProxyHeadersMiddleware with an explicit trusted proxy list.
  • Never trust client-supplied IP headers directly without proxy validation.

F-19 · Missing Content Security Policy (CSP)

IDF-19
SeverityLow
OWASP CategoryA03:2021 – Injection (XSS)
Locationfrontend/next.config.js

Description

No Content Security Policy header was identified. CSP is a critical defence-in-depth measure that instructs browsers which sources of scripts, styles, images, and other resources are legitimate. Without CSP, a successful XSS attack can freely load attacker-controlled scripts, exfiltrate tokens, and manipulate the UI.

How to Fix
  • Add a CSP header in next.config.js via the headers() function.
  • Start with a reporting-only policy (Content-Security-Policy-Report-Only) to identify violations before enforcing.
  • A baseline CSP for Next.js: default-src 'self'; script-src 'self' 'nonce-{nonce}'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' wss: https:;
  • Use nonce-based CSP for inline scripts generated by Next.js.

F-20 · Admin "All" Permission Wildcard

IDF-20
SeverityInformational
OWASP CategoryA01:2021 – Broken Access Control
Locationbackend/app/core/permissions.py

Description

The RBAC system grants administrators a special 'all' permission that bypasses all permission checks. While practical, this means there is no way to create a "restricted admin" (e.g., a content moderator with admin access only to moderation endpoints). IDF-21

The application uses python-socketio 5.11.0 for real-time WebSocket communication. The authentication mechanism for WebSocket connections (whether JWT is validated on connection, and whether per-event authorisation is enforced) was not fully visible in the reviewed code.   • Implement per-room / per-channel authorisation checks.
  • Document the Socket.IO authentication flow in the security documentation.

F-22 · Database Connection String in Environment Variable

IDF-22
SeverityInformational
OWASP CategoryA05:2021 – Security Misconfiguration
Locationbackend/app/core/config.py – DATABASE_URL

Description

The database connection string (which includes username, password, host, and database name) is passed as a single DATABASE_URL environment variable. This is a standard pattern, but it means that any process or log that dumps environment variables exposes full database credentials.

How to Fix
  • Use a secrets manager (AWS Secrets Manager, HashiCorp Vault) to inject the DATABASE_URL at runtime rather than baking it into the container environment.
  • Ensure application logs do not print environment variables or startup configuration.
  • Rotate database credentials periodically; use IAM-based authentication (AWS RDS IAM) where possible to avoid static passwords.

6. Conclusion & Remediation Priority

The MyHigh5 platform is a feature-rich, architecturally sound application. The use of Pydantic for input validation, bcrypt for password hashing, SQLAlchemy ORM for parameterised queries, and HMAC signature verification for webhooks demonstrates a security-conscious development approach.

However, two Critical findings must be addressed before any production launch:

  1. F-01 (Hardcoded JWT Secret) — Any developer or repository reader can currently forge admin tokens. Remove the default and generate a strong random secret.
  2. F-02 (Committed Secrets) — Rotate all exposed API keys immediately and purge them from git history. Add .env* to .gitignore.
