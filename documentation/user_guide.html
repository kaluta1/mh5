🏆 Project Overview

MyHigh5 is a multilingual, feature-rich online contest and voting platform. It enables individuals and organisations to create competitions, register contestants, manage voting rounds, process payments (fiat and cryptocurrency), verify identities via KYC, and run affiliate marketing campaigns — all under a single, unified platform.

What is MyHigh5?

MyHigh5 is designed as a global contest platform where:

  • Contest Organisers create and manage competitions with configurable rules, voting types, entry fees, and prize structures.
  • Contestants register, upload media (photos/videos), undergo identity verification, and compete for prizes and rankings.
  • Voters discover, follow, and vote for their favourite contestants.
  • Affiliates earn commissions through a 10-level multi-tier referral system.
  • Advertisers reach the audience through an integrated sponsor/advertising system (Annual Ads).
  • Administrators manage the entire platform: users, contests, KYC reviews, financial reports, and content moderation.

Request Lifecycle

  1. Client sends HTTP request with Authorization: Bearer <JWT> header.
  2. CORS Middleware validates the request origin against the allowlist.
  3. FastAPI Router matches the route and calls the endpoint function.
  4. Dependency Injection (get_current_user) decodes and validates the JWT, loads the user from PostgreSQL.
  5. RBAC Check verifies the user has the required permission for the endpoint.
  6. Pydantic validates the request body/query params against the schema.
  7. Business Logic executes, querying PostgreSQL via SQLAlchemy ORM.
  8. Background Tasks (if needed) are dispatched to Celery via Redis.
  9. Response is serialised by Pydantic and returned as JSON.

🔒 Authentication & Security

Authentication Flow

User Registration

User submits email, password, and optional sponsor code to POST /api/v1/auth/register. Password is hashed with bcrypt (12 rounds). A verification email is sent via Resend.

Email Verification

User clicks the link in the verification email. Token is valid for 24 hours. Account becomes active after verification.

Login

User submits credentials to POST /api/v1/auth/login (OAuth2 password form). Server validates password hash, generates a JWT token (HS256, 7-day expiry), and returns it.

Authenticated Requests

Client includes Authorization: Bearer <token> header on all subsequent requests. The get_current_active_user dependency validates the token on every request.

Role-Based Access Control (RBAC)

The platform uses a hierarchical RBAC system:

  • Roles are assigned to users and can inherit from parent roles.
  • Permissions are string-named capabilities (e.g., manage_contests, approve_kyc).
  • Admin roles have an 'all' permission that bypasses individual permission checks.
  • Endpoint protection uses PermissionChecker dependency: require_permission("manage_contests").
# Example dependency usage in an endpoint:
@router.post("/admin/contests")
async def create_admin_contest(
    db: Session = Depends(get_db),
    current_user: User = Depends(require_permission("manage_contests"))
):

📄 Database Migrations
Automatic: Migrations run automatically when the backend starts via alembic upgrade heads in the startup lifespan handler.

Manual Migration Commands

cd D:\laragon\www\mh5\backend

# Apply all pending migrations:
alembic upgrade heads

# Check current revision:
alembic current

# View migration history:
alembic history --verbose

# Create a new migration (after model changes):
alembic revision --autogenerate -m "Add new_field to users"

# Roll back one revision:
alembic downgrade -1

# Roll back to base (CAUTION — drops all tables):
alembic downgrade base
Warning: Never run alembic downgrade base against a production database without a full backup.
🕑 Celery Background Workers

Celery handles asynchronous tasks such as sending emails, processing payments, and running scheduled jobs. Redis is used as both the broker and result backend.

Start a Celery Worker

cd D:\laragon\www\mh5\backend

# Activate venv first, then:
celery -A app.worker.celery_app worker --loglevel=info

# Windows (use --pool=solo or --pool=gevent):
celery -A app.worker.celery_app worker --loglevel=info --pool=solo

Start Celery Beat (Scheduled Tasks)

celery -A app.worker.celery_app beat --loglevel=info
Background Schedulers: The FastAPI lifespan also starts in-process schedulers for: payment processing, contest status updates, season migration, and monthly round creation. These run within the main process and do not require separate Celery beat for basic functionality.
